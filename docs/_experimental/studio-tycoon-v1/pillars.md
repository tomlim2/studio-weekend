# Design Pillars

타협 불가한 설계 원칙. 피처가 이 원칙과 충돌하면 피처가 진다.

각 pillar는 "actionable test"로 판정 가능해야 함 — "이 기능이 pillar에 부합하는가?"를 팀원(또는 AI 에이전트)이 10초 안에 대답할 수 있는 수준.

---

## 1. 토큰 경제가 모든 시스템의 중심축이다

**원칙:** 플레이어가 하는 모든 의미 있는 결정은 결국 "토큰을 어디에 쓸 것인가"로 환원된다. 토큰이 뇌동맥이다.

**Why:** 리서치 결과 이게 우리 게임의 진짜 novel 축. 이거 약해지면 "또 하나의 타이쿤"으로 전락.

**Actionable test:**
> 새 피처가 **토큰 소비/획득/효율**에 관여하지 않는다면 → 왜 추가하는지 방어해야 한다.

**적용 예:**
- 에이전트 고용: 토큰 고정비 (월세 개념)
- 태스크 실행: 토큰 가변비
- 휴식/컴팩트: 토큰 소비 없이 성능 회복
- 업그레이드: 토큰 효율 개선

**반례 (추가 금지):**
- 토큰과 무관한 "평판 포인트" 별도 시스템
- 장식용 꾸미기 메뉴 (비용 없는 코스메틱)

---

## 2. 풍자는 애정에서 나온다

**원칙:** AI 업계·개발자 문화에 대한 조크는 **사랑이 깔린 메타 개그**다. 냉소·조롱으로 기울면 안 된다.

**Why:** 타겟 청중 = 실제 AI 툴 쓰는 개발자. 본인 일상을 조롱당한다고 느끼면 이탈. *Papers Please*가 전체주의를 다루면서도 애정 있게 인간을 그리는 톤.

**Actionable test:**
> 이 대사/이벤트가 **실제 Claude Code 유저가 읽고 "맞아 이거 나야 ㅋㅋ" 하고 공감하는가**, 아니면 **"아 이거 AI 싫어하는 사람이 쓴 거네" 반응이 나오는가**?

**적용 예:**
- 환각 이벤트: "에이전트가 함수 3개를 상상해서 import했습니다" → 웃김, 공감
- 번아웃: "수석 엔지니어가 휴가 신청서를 제출했습니다" → 애정
- 모델 업데이트: "Opus 4.8 출시! 직원들이 연봉 인상 요구합니다" → 메타 개그

**반례 (피하기):**
- "AI는 결국 인간을 대체합니다" 같은 묵시록 톤
- 에이전트를 무능/위험한 존재로만 묘사

---

## 3. 결과의 원인이 항상 보인다 (Decision Legibility)

**원칙:** 플레이어는 "왜 이 게임이 30점을 받았는가"를 UI에서 즉각 추적할 수 있어야 한다. 시스템 내부가 투명해야 한다.

**Why:** 리서치에서 확인된 **장수 게임 #1 이탈 사유**: "내 결정이 왜 이 결과로 이어졌는지 모르겠다." *Software Inc.*의 가장 큰 약점.

**Actionable test:**
> 플레이어가 결과 화면에서 **숫자/평점 각 줄에 대해 "왜?" 라고 클릭했을 때 원인 사슬을 볼 수 있는가**?

**적용 예:**
- 게임 출시 점수 = 디자인 40% + 코드 30% + 아트 20% + 운 10% → 각 항목 expand 가능
- "디자인 40점 중 22점" 클릭 → 어느 디자이너가 어느 태스크 수행 → 어느 토큰 소비 → 어떤 환각 이벤트 발생
- 에이전트 성능 저하 = 컨텍스트 로트 + 번아웃 + 모델 다운그레이드 → 각 원인 기여도 표시

**반례 (피하기):**
- 블랙박스 점수 ("운이 좋았다" 식 설명 회피)
- 히든 스탯 (플레이어에게 안 보이는 내부 값)

---

## 4. 모더 퍼스트 (Data-Driven From Day 1)

**원칙:** 에이전트 정의, 이벤트, 시나리오, 모델 카드 — 모두 **JSON/YAML 파일로 분리**해서 처음부터 외부 편집 가능하게. 코드에 하드코딩 금지.

**Why:** 리서치 결과 **타이쿤 장수의 #1 레버 = 모딩**. 나중에 리팩터하려면 비용 폭발. 처음부터 데이터 드리븐.

**Actionable test:**
> 이 피처의 **컨텐츠**가 `.json`/`.yaml`로 빠져나올 수 있는가? 안 된다면 지금 당장 빠져나오게 설계하라.

**적용 예:**
- 에이전트 카드: `data/agents/creative-director.json`
- 이벤트 테이블: `data/events/hallucination.json`
- 모델 릴리스 타임라인: `data/timeline/models.json`
- 시나리오 (시작 상태): `data/scenarios/indie-solo.json`

**반례 (금지):**
- `if agent.name == "opus-director":` 같은 하드코딩
- 이벤트 트리거를 GDScript에 직접 작성
- 밸런스 값 소스 파일 안에 const로 박기

## 5. 30분 안에 한 판 끝난다

**원칙:** 한 "스튜디오 생애" (창업 → 도산 또는 명작 출시)가 **30~45분 안에 끝난다**. 리플레이가 전략.

**Why:** 리서치 확인: 주중 저녁 1~2시간 짬 플레이 청중 노림. *Slay the Spire* 모델. 20시간 세이브 파일 끌고 가는 대형 타이쿤은 우리가 아님.

**Actionable test:**
> 새 메카닉이 **한 판 시간을 늘리는가**? 늘린다면 **깊이**를 추가하는가 **느림**을 추가하는가?

**적용 예:**
- 루프 단위: 1 스프린트 = 실시간 30초~1분
- 스튜디오 수명: 12~20 스프린트
- 메타-진행: 새 스튜디오 시작할 때마다 언락된 에이전트·시나리오 풀 확장

**반례 (피하기):**
- 세이브/로드 필수 장편 캠페인 (그건 다른 게임)
- 단일 스프린트 안에서 지루한 대기 시간

## Pillar Priority When Conflicting

충돌 발생 시 우선순위:

```
1. 토큰 경제 중심축     ← 흔들리면 게임 정체성 소멸
2. 결과의 원인 가시성   ← 흔들리면 유저 이탈
3. 30분 한 판           ← 흔들리면 타겟 청중 놓침
4. 모더 퍼스트          ← 흔들리면 장수 못 함
5. 애정 담긴 풍자       ← 흔들리면 톤 깨짐 (톤은 이터레이션 가능)
```

> 톤(#5)이 마지막인 이유: 톤은 대사/아트로 후반 수정 가능. 구조(#1~#4)는 후반 수정 비용 기하급수적.
